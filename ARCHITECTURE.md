# Foro System Architecture

This document is the top-level architecture summary for the current codebase. It is intentionally shorter and more maintainable than a file-by-file dump. Use it to understand boundaries first, then drill into `docs/architecture/` or source files.

## Overview

Foro is made of three cooperating parts:

1. A React 19 frontend built with Vite
2. An Express server that proxies upstream services, serves static builds, and exposes state and billing APIs
3. A VitePress docs site shipped alongside the application

## High-Level Flow

```text
User interaction
  -> React workspace state
  -> hooks and services
  -> internal /api routes
  -> upstream providers or server-side store
  -> normalized result back into app state
  -> persisted UI and rendered workspace output
```

## Frontend Architecture

Main frontend modules:

- `src/App.tsx`
- `src/components/AppWorkspaceRouter.tsx`
- `src/hooks/useHomeFeedWorkspace.ts`
- `src/hooks/useSearchWorkspace.ts`
- `src/hooks/useAudienceSearch.ts`
- `src/hooks/useBilling.ts`
- `src/lib/persistence/client.ts`

Responsibilities:

- hold cross-workspace state
- coordinate plan-gated behavior
- manage browser-history-backed UI state restoration
- persist lightweight and collection state through shared adapters
- render six top-level workspaces

Current workspace set:

- `home`
- `content`
- `read`
- `audience`
- `bookmarks`
- `pricing`

## Server Architecture

Main backend modules:

- `server/app.cjs`
- `server.cjs`
- `server/lib/config.cjs`
- `server/lib/appStateStore.cjs`
- `server/lib/articleExtractor.cjs`

Server responsibilities:

- authenticate internal API calls with `x-internal-token`
- proxy X/Twitter, xAI, and Tavily traffic
- expose backend persistence through `/api/state/:namespace/:key`
- fetch and extract readable article content
- create and inspect Stripe checkout sessions
- serve built app output at `/test`
- serve built docs output at `/test/docs`

## Persistence Model

The app supports two practical persistence strategies:

- browser persistence via local storage and IndexedDB-backed hooks
- backend persistence via the state API and server-side store implementations

This keeps feature logic mostly independent from storage location.

## External Integrations

Current integration points:

- X/Twitter upstream through `/api/twitter/*`
- xAI through `/api/xai/*`
- Tavily through `/api/tavily/search`
- RSS fetch proxy through `/api/rss`
- article extraction through `/api/article`
- Stripe checkout through `/api/billing/*`

## Documentation Architecture

Documentation is part of the product workflow, not a side artifact.

- source docs live under `docs/`
- generated docs data lives under `docs/.vitepress/data`
- docs status, changelog, and draft suggestions are generated by scripts
- docs are built and served together with production app output

## Product Rules That Matter Architecturally

- feed visibility is plan-limited
- AI filter scope follows visible feed scope
- RSS duplicate suppression is durable during sync
- clearing the Home feed intentionally resets RSS seen-state
- article reading reuses cached Thai translation when available
